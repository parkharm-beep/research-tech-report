# Git / GitHub

## 1. 개요 (Overview)
- **기술명:** Git / GitHub
- **정의:** Git은 파일의 변경 이력을 추적·관리하는 분산 버전 관리 시스템이고, GitHub는 Git 저장소를 클라우드에서 호스팅·협업하는 플랫폼
- **배경:** "내가 뭘 바꿨는지 모르겠다", "예전 버전으로 돌아가고 싶다", "팀원과 코드를 합치다 충돌났다"는 문제를 해결하기 위해 등장했다. 비개발자에게는 **용량 제한 없는 무료 구글 드라이브 + 타임머신** 정도로 이해하면 된다.

## 2. Git vs GitHub 차이

| | Git | GitHub |
|---|---|---|
| **정의** | 버전 관리 도구 (소프트웨어) | Git 저장소 호스팅 플랫폼 (웹서비스) |
| **비유** | 데이터베이스 서버 자체 | 그 DB에 접근하는 GUI 클라이언트 |
| **유료 여부** | 무료 (오픈소스) | 기본 무료, 일부 기능 유료 |
| **대안** | - | GitLab, Bitbucket, Git Desktop 등 |

## 3. 핵심 개념 (Key Concepts)

### 저장소 (Repository, Repo)
프로젝트 파일과 전체 변경 이력이 저장되는 공간. 로컬(내 컴퓨터)과 원격(GitHub)으로 나뉨.

### 커밋 (Commit)
변경 사항을 기록하는 스냅샷. 언제·누가·무엇을 바꿨는지 메시지와 함께 저장. 타임머신의 저장 포인트.

### 브랜치 (Branch)
원본(main)을 건드리지 않고 새 기능을 개발하는 독립적 작업 공간. 실험 후 검토를 거쳐 main에 합침(Merge).

### 롤백 (Rollback)
커밋 ID를 기준으로 이전 버전으로 되돌리기. 카카오톡이 잘못된 업데이트를 되돌릴 수 있는 것도 이 원리.

### 풀 리퀘스트 (Pull Request, PR)
내 브랜치의 변경 사항을 main에 합쳐달라는 요청. 코드 리뷰·토론 후 병합.

## 4. 기본 워크플로우

```bash
git clone {저장소 URL}        # 원격 저장소를 내 컴퓨터에 복제
git add {파일}                # 변경 사항 스테이징
git commit -m "변경 내용"     # 스냅샷 저장
git push                      # 원격(GitHub)에 업로드
git pull                      # 원격 최신 변경 사항 내려받기
```

## 5. 왜 Git을 써야 하는가
1. **히스토리 추적:** 언제·누가·무엇을 바꿨는지 전체 이력 보존
2. **롤백:** 잘못된 변경 사항을 특정 시점으로 즉시 되돌리기
3. **협업:** 여러 사람이 동시에 작업 후 안전하게 합치기
4. **AI 코딩 연동:** Claude Code, Codex 등 AI 코딩 도구가 Git 기반으로 변경 추적·PR 생성

## 6. 장단점 (Pros & Cons)
### 장점 (Pros)
- 무한 버전 이력 보존 (무료)
- 실수해도 이전 버전으로 즉시 복구
- 오픈소스 생태계의 표준 — 전 세계 개발자와 코드 공유 가능
### 단점 (Cons)
- 초보자에게 영어 인터페이스·명령어가 낯설어 진입장벽 존재
- 병합 충돌(Merge Conflict) 발생 시 수동 해결 필요
