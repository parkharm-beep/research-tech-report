# 소프트웨어 아키텍처 (Software Architecture)

## 1. 개요 (Overview)
- **기술명:** 소프트웨어 아키텍처 (Software Architecture)
- **정의:** 소프트웨어 애플리케이션을 구성하는 프론트엔드·백엔드·데이터베이스의 3계층 구조와 그 흐름을 정의하는 설계 개념
- **배경:** 비개발자도 AI 코딩 도구로 소프트웨어를 만드는 시대가 됐지만, 구조를 모르면 올바른 설계 지시를 할 수 없다. 식당 비유(홀·주방·냉장고)로 이해하는 것이 가장 직관적이다.

## 2. 3계층 구조 (Three-tier Architecture)

| 계층 | 별칭 | 역할 | 식당 비유 |
|---|---|---|---|
| **프론트엔드** | 클라이언트 | 사용자와 직접 상호작용 — 화면·버튼·메뉴 | 홀, 테이블, 메뉴판 |
| **백엔드** | 서버 | 사용자 요청을 받아 처리하고 DB와 통신 | 주방, 요리사 |
| **데이터베이스** | DB | 데이터를 영구 저장하고 꺼내줌 | 냉장고, 식재료 보관소 |

데이터 흐름: `사용자 → 프론트엔드 → (요청) → 백엔드 → (쿼리) → DB → (데이터 반환) → 백엔드 → (응답) → 프론트엔드 → 사용자`

## 3. 주요 개념 (Key Concepts)

### 라우팅 (Routing)
사용자를 원하는 페이지로 보내주는 이정표. 식당에서 "2인석으로 안내해 드릴게요"와 같은 역할.

### 서빙 (Serving)
사용자의 요청 데이터를 백엔드로 전달하는 과정. 홀 종업원이 주문을 받아 주방에 전달하는 것.

### API (Application Programming Interface)
프론트엔드와 백엔드가 통신하는 규약·인터페이스. 종업원이 주방과 대화하는 주문서 양식.

### 서버리스 (Serverless)
별도의 백엔드 서버 없이 프론트엔드에서 직접 DB·클라우드 함수와 통신하는 구조. 단순한 앱에 효율적이지만, 복잡한 보안·로직이 필요한 경우 전통적 백엔드가 적합.

## 4. 공학적 원칙
- **정답은 여러 개:** 목적에 따라 최적 아키텍처가 달라진다 (서버리스 vs 풀스택, 마이크로서비스 vs 모놀리식)
- **확장성:** 사용자 수·데이터가 늘어도 시스템이 견딜 수 있게 설계
- **보안:** 백엔드가 DB를 직접 노출하지 않고 중간에서 인증·인가 처리
