# 벡터 데이터베이스 (Vector Database)

## 1. 개요 (Overview)
- **기술명:** 벡터 데이터베이스 (Vector Database)
- **정의:** 텍스트·이미지 등을 임베딩 모델로 변환한 고차원 벡터를 저장하고, 질의 벡터와 가장 의미적으로 유사한 항목을 빠르게 검색(ANN Search)하는 데이터베이스
- **배경:** 기존 SQL 기반 키워드 검색은 의미(Semantic) 기반 검색이 불가능하다. LLM 시대 AI는 파일 경로가 아닌 의미로 데이터를 접근하기 때문에, 벡터 DB가 AI 데이터 인프라의 핵심으로 부상했다.

## 2. 주요 특징 (Key Features)
- 각 문서/문단을 임베딩 모델로 수천 차원의 벡터로 변환하여 저장
- HNSW(Hierarchical Navigable Small World) 등 그래프 기반 인덱스로 ANN 검색 최적화
- 전체 벡터를 순차 비교하는 대신 인덱스 구조 탐색으로 연산량 수십~수백 배 절감
- 핫(hot) 벡터는 메모리, 콜드(cold) 벡터는 SSD/NVMe에 계층 저장
- 에이전틱 AI 시대에 에이전트의 외부 지식(External Knowledge) 저장소로 핵심 역할

## 3. 작동 원리 (How it Works)
**인덱싱 (사전 준비):**
1. 문서를 청킹(Chunking)으로 단락 단위 분할
2. 임베딩 모델로 각 청크를 벡터 변환
3. HNSW 등 그래프 인덱스로 벡터 저장·구조화

**검색 (Query):**
1. 질의 텍스트를 동일 임베딩 모델로 벡터 변환
2. 인덱스 그래프를 탐색하여 Top-K 최근접 벡터 선별
3. 해당 벡터가 가리키는 원본 문서 청크 반환 → LLM 컨텍스트에 주입

## 4. 주요 활용 사례 (Use Cases)
- 사례 1: RAG 파이프라인 — 기업 내부 문서를 벡터화하여 LLM 응답 근거로 활용
- 사례 2: 에이전트 장기 기억 — 에이전트가 수행한 이력을 벡터 DB에 저장하여 후속 작업에 검색
- 사례 3: 스토리지 시맨틱 인터페이스 — 클라우드 스토리지 전체를 벡터화하여 자연어로 검색

## 5. 장단점 (Pros & Cons)
### 장점 (Pros)
- 의미 기반 시맨틱 검색으로 키워드 불일치에도 관련 문서 검색
- RAG, 에이전트 등 AI 파이프라인과 자연스럽게 통합
- 대규모 문서 처리를 위한 분산 아키텍처 지원
### 단점 (Cons)
- 에이전틱 AI 확산으로 검색 워크로드가 폭증 → CPU 병목 심화 (VDPU 등 전용 가속기 필요성 대두)
- 임베딩 모델 품질에 검색 정확도 종속
- 정확한 키워드 매칭에서는 전통 검색(BM25)보다 불리 → 하이브리드 검색 권장
