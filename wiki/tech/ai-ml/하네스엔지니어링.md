# 하네스 엔지니어링 (Harness Engineering)

## 1. 개요 (Overview)
- **기술명:** 하네스 엔지니어링 (Harness Engineering)
- **정의:** AI 에이전트(Claude Code, Codex 등)가 최적의 출력을 낼 수 있도록, 충분한 컨텍스트(기획·디자인 레퍼런스·제약 조건 등)를 미리 설계하고 제공하는 엔지니어링 접근법
- **배경:** AI 바이브코딩이 확산되면서 "AI에게 어떻게 지시하느냐"보다 "AI가 일할 수 있는 환경(하네스)을 어떻게 갖추느냐"가 결과 품질의 핵심 변수가 됨. 프롬프트 엔지니어링의 확장 개념이다.

## 2. 주요 특징 (Key Features)
- **컨텍스트 사전 설계:** 요청 전에 AI가 참조할 디자인 MD, 레퍼런스, 컬러 팔레트, 기획서 등을 준비
- **품질의 책임은 사람에게:** AI 출력물이 나쁜 것은 AI 성능 문제가 아니라 충분한 자료를 제공하지 않은 것
- **워크플로우 재설계:** AI를 단순 도구가 아닌 팀원으로 보고, AI가 처리할 업무를 미리 구조화
- **에러 처리 포함:** 에러 메시지를 AI에게 그대로 전달하여 해결책 도출 → 비개발자도 가능

## 3. 작동 원리 (How it Works)
하네스 엔지니어링의 기본 사이클:
1. **기획·설계 단계:** 만들 것을 명확히 정의 (무엇을 만들지 모르면 AI도 설계 불가)
2. **컨텍스트 준비:** 디자인 MD, 색상 기준, 레퍼런스 앱, API 명세 등 수집
3. **AI에게 전달:** 준비된 컨텍스트와 함께 작업 지시
4. **결과 검토 및 피드백:** 출력이 기대와 다르면 누락된 컨텍스트를 추가 후 재시도
5. **반복 개선:** 워크플로우 자체를 지속적으로 최적화

핵심 원칙: **"AI가 멍청한 게 아니라, 내가 충분한 자료를 주지 않은 것이다"**

## 4. 주요 활용 사례 (Use Cases)
- 사례 1: 디자인 MD 제공 후 Claude + Figma MCP로 UI 자동 생성
- 사례 2: ERD·API 스펙 문서 제공 후 백엔드 코드 자동 생성
- 사례 3: 기존 코드베이스를 컨텍스트로 제공하여 일관된 스타일 유지하며 기능 추가

## 5. 장단점 (Pros & Cons)
### 장점 (Pros)
- 사전 준비가 충실할수록 AI 출력 품질이 극적으로 향상
- 비개발자도 체계적 접근으로 전문 개발자 수준의 결과물 획득 가능
- AI 에러 처리를 AI 자체에 위임 → 빠른 반복 개발
### 단점 (Cons)
- 초기 컨텍스트 준비 비용(시간·노력) 높음
- 컨텍스트 품질 판단 능력 자체가 일정 수준의 도메인 지식을 요구
