# 하네스 엔지니어링 (Harness Engineering)

## 1. 개요 (Overview)
- **기술명:** 하네스 엔지니어링 (Harness Engineering)
- **정의 (비개발자 관점):** AI 에이전트가 최적의 출력을 낼 수 있도록 충분한 컨텍스트(기획·디자인·제약 조건 등)를 사전에 설계·제공하는 엔지니어링 접근법
- **정의 (개발자/시스템 관점):** 장기 작업을 수행하는 AI 에이전트의 **상태(State)·메모리(Memory)·실행 흐름(Execution Flow)**을 안정적으로 관리하는 운영 시스템
- **배경:** 단발성 프롬프트의 한계(맥락 끊김, 복잡도 붕괴, 반복 개입)를 극복하기 위해, AI 에이전트를 팀처럼 조직하고 상태와 규칙을 시스템 수준에서 관리하는 방식으로 발전했다.

## 2. 두 가지 접근 관점

### 관점 A: 컨텍스트 설계 (비개발자·바이브코딩)
- AI 출력 품질은 제공한 컨텍스트의 품질에 비례
- 디자인 MD, 레퍼런스, API 명세, 제약 조건을 미리 준비
- 핵심 원칙: **"AI가 멍청한 게 아니라, 내가 충분한 자료를 주지 않은 것이다"**

### 관점 B: 에이전트 운영 시스템 (개발자·시스템 구축)
- 프롬프트→결과→오류→재프롬프트의 단발 루프를 탈피
- 오케스트레이터가 멀티 에이전트를 통제하고 상태를 지속 관리
- 에이전트들이 살아 있는 상태로 유지되며 메시지를 주고받음([[팀모드]])

## 3. 하네스 3가지 유형

| 유형 | 설명 | 예시 |
|---|---|---|
| **로컬 하네스** | 훅·스킬을 처음부터 직접 구현 | CLAUDE.md 기반 커스텀 훅 |
| **오픈소스 하네스** | 기존 프레임워크 활용 | Claude Code (GStack), Codex, SuperPowers, Ouroboros |
| **하이브리드** | 프레임워크 + 로컬 메모리 파일 조합 | 오픈소스 + 자체 memory.md/docs 구조 |

## 4. 핵심 구성 요소

### 팀모드 (Team Mode)
에이전트들이 살아 있는 상태로 유지되며 서로 메시지를 주고받고 병렬로 역할을 분담. 단순 분리 실행이 아닌 **에이전트 간 소통**이 핵심. → [[팀모드]] 참조

### 메모리 시스템
```
memory.md           ← 전체 실패 토픽 인덱스
memory/
  daily/            ← 일별 실패 로그 누적
  topics/           ← 3회 이상 반복 에러 승격 → 항상 주입
```
- 실패 → 원인 정리 → 다음 작업 주입 → 재사용 (에이전트의 장기 기억)

### Git Worktrees
- 에이전트마다 독립된 작업 폴더(워크트리) 부여
- 브랜치 스위칭·스태시 없이 병렬 작업 가능
- 작업 완료 후 머지·리베이스로 통합

### Tmux (세션 관리)
- 에이전트 세션을 분리·유지
- SSH가 끊겨도 백그라운드에서 세션 유지
- 하나의 터미널에서 다수 세션 동시 관리

## 5. 실무 워크플로우

```
프롬프팅 (ChatGPT/Claude/Gemini로 교차검증)
  ↓
딥 인터뷰 (질문 먼저 → 합의된 계획 수립)
  ↓
팀 모드 실행 (오케스트레이터 → 에이전트 분배)
  ↓
독립 워크트리에서 병렬 작업
  ↓
결과 검증 (울트라 QA / 사람 검토)
  ↓
메인 브랜치 머지
  ↓
실패 메모리 정리
```

### 딥 인터뷰 (Deep Interview)
플랜 모드의 고도화 버전. 바로 만들지 않고 먼저 질문하여 모호성을 낮추고, 합의된 계획만 실행. POC 단계에서 여러 프로젝트(01, 02...)를 병렬 시도 후 최적안 선택.

## 6. 개발자의 역할 변화

| 과거 | 현재 (하네스 시대) |
|---|---|
| 코드 직접 작성 | 방향 설정 (에이전트로 리서치) |
| 세부 구현 | 규칙 정의 (docs/architecture.md) |
| 수정 반복 | 결과 검증 (QA 에이전트 활용) |

**개발자는 디렉터.** AI는 자동 도구가 아니라 가르쳐야 할 시스템. 완벽하게 맡기는 순간 통제가 무너진다.

## 7. 장단점 (Pros & Cons)
### 장점 (Pros)
- 단발 프롬프트 한계(맥락 끊김) 극복 → 장기 복잡 작업 수행 가능
- 실패 메모리로 반복 오류 방지
- 병렬 에이전트로 작업 속도 향상
### 단점 (Cons)
- 초기 하네스 설계 및 메모리 구조 구축 비용
- 에이전트 간 충돌·머지 복잡성
- 통제 없이 위임하면 오히려 품질 저하
