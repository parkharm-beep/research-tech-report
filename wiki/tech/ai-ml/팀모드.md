# 팀모드 (Team Mode)

## 1. 개요 (Overview)
- **기술명:** 팀모드 (Team Mode)
- **정의:** [[하네스엔지니어링]]의 핵심 실행 방식으로, 오케스트레이터 에이전트가 복수의 워커 에이전트에게 역할을 분배하고, 에이전트들이 서로 메시지를 주고받으며 병렬로 작업을 수행하는 다중 에이전트 협업 구조
- **배경:** 단일 에이전트는 긴 작업에서 맥락이 끊기고 복잡도가 붕괴되는 한계가 있다. 팀모드는 인간 팀처럼 AI 에이전트들을 조직하여 장기·복잡 작업을 안정적으로 수행한다.

## 2. 핵심 구조

```
오케스트레이터
  ├─ 에이전트 A (기능 X 담당, worktree-a/)
  ├─ 에이전트 B (기능 Y 담당, worktree-b/)
  └─ 에이전트 C (기능 Z 담당, worktree-c/)
       ↓
  [검증 단계]
       ↓
  메인 브랜치 머지
```

## 2. 주요 특징 (Key Features)
- **에이전트 생존 유지:** 각 에이전트가 작업 완료 후 종료되지 않고 살아 있는 상태로 유지
- **메시지 기반 소통:** 에이전트끼리 작은 단위로 메시지를 주고받으며 진행 상황 공유
- **독립 워크트리:** 각 에이전트는 Git worktree로 분리된 독립 작업 폴더를 가짐 → 브랜치 충돌 없음
- **오케스트레이터 통제:** 작업 분배, 흐름 제어, 최종 머지를 오케스트레이터가 담당
- **Tmux 세션 관리:** 각 에이전트를 독립 Tmux 세션으로 관리, SSH 끊김에도 세션 유지

> **단순 병렬 실행과의 차이:** "에이전트 여러 개에 나눠서 3일 뒤 머지"는 팀모드가 아님. 에이전트들이 실시간으로 소통하며 충돌을 사전에 해소하는 것이 팀모드의 핵심.

## 3. 작동 원리 (How it Works)
1. **오케스트레이터**가 사용자 요청을 받아 작업 분석
2. 역할별 에이전트에게 작업 분배 (각자 독립 worktree 할당)
3. 에이전트들이 **Tmux 세션에서 병렬 실행** (살아 있는 상태 유지)
4. 에이전트 간 메시지 교환으로 의존성·충돌 실시간 해소
5. 작업 완료 시 **검증 단계** 통과 후 메인 브랜치에 머지
6. 실패 내역은 **메모리 시스템**에 자동 기록

## 4. 주요 활용 사례 (Use Cases)
- 사례 1: 대규모 코드베이스 리팩터링 — 모듈별로 에이전트를 분배하여 병렬 수정 후 통합
- 사례 2: 풀스택 앱 개발 — 프론트엔드·백엔드·DB 에이전트가 동시 개발하며 API 계약 소통
- 사례 3: 멀티 채점 서버 POC — 동일 문제를 다른 방식으로 구현한 여러 버전을 병렬 생성 후 최적안 선택

## 5. 장단점 (Pros & Cons)
### 장점 (Pros)
- 장기 복잡 작업에서 단일 에이전트 대비 안정성 대폭 향상
- 병렬 처리로 작업 시간 단축
- 실패 메모리 누적으로 반복 오류 방지
### 단점 (Cons)
- 하네스 설계(오케스트레이터 로직, worktree 구조)의 초기 구축 비용
- 에이전트 간 충돌 발생 시 머지 복잡도 증가
- 통제 없이 위임하면 오히려 품질 저하
