# 파인튜닝 (Fine-tuning)

## 1. 개요 (Overview)
- **기술명:** 파인튜닝 (Fine-tuning)
- **정의:** 사전학습된 LLM의 가중치를 특정 도메인·태스크에 맞게 추가 학습시켜, 해당 영역에서의 성능을 높이는 기법
- **배경:** GPT·Claude 같은 범용 LLM은 회사 고유 명사, 사내 데이터, 보안 민감 정보를 처리하기 어렵다. 파인튜닝을 통해 공개 클라우드 LLM 없이 사내에서 특화 모델을 운영할 수 있다.

## 2. 파인튜닝이 필요한 이유 (Why Fine-tune)
1. **민감 데이터 보안:** 외부 LLM API에 보낼 수 없는 고객·의료·금융 데이터를 로컬 모델로 처리
2. **도메인 특화:** 범용 LLM이 모르는 사내 전문 용어·제품명·약어 등을 학습시켜 정확도 향상
3. **비용 최적화:** 하루 수만 건 배치 처리 시 클라우드 API 비용보다 로컬 파인튜닝 모델이 경제적
4. **응답 형식 통제:** 특정 JSON 스키마·출력 포맷을 항상 준수하도록 강제 ([[출력 계약]])

## 3. 주요 기법 (Key Techniques)

### 양자화 (Quantization)
모델 가중치의 정밀도를 낮춰 크기·메모리 축소
- **PTQ (Post-Training Quantization):** 학습 완료 후 양자화 적용 — 쉽지만 성능 저하 가능성
- **QAT (Quantization-Aware Training):** 학습 단계에서 양자화 적용 — 더 좋지만 복잡
- **실무 표준:** 8비트(INT8) 가장 보편적, 4비트는 성능 손실 크므로 주의
- → [[양자화]] 참조

### PEFT / LoRA (Parameter Efficient Fine-Tuning)
전체 가중치를 재학습하지 않고 소수의 어댑터 파라미터만 학습
- 원본 가중치를 동결(freeze)하고 저차원 어댑터 행렬(A, B)만 학습
- 16GB 맥북에서도 8B 모델 파인튜닝 가능
- → [[LoRA]] 참조

## 4. 실무 파인튜닝 워크플로우 (Practical Workflow)
1. **베이스 모델 선택:** Qwen, Llama, Mistral 등 오픈소스 Instruct 모델 선택
2. **출력 계약 설계:** 추출할 JSON 스키마, 허용값, 예외 처리 규칙 정의 ([[출력 계약]])
3. **학습 데이터 구성:** (입력 텍스트, 기대 출력 JSON) 쌍으로 구성된 수백~수천 건 데이터셋
4. **양자화 + LoRA 파인튜닝:** Unsloth(NVIDIA GPU) 또는 MLX(Apple Silicon) 활용
5. **평가 및 배포:** 할루시네이션 발생률 측정 → 임계치 이하 시 Ollama 등으로 서빙

## 5. LLM 운영 고려사항 (Operational Concerns)
파인튜닝된 모델은 "코드"가 아닌 **운영해야 할 시스템**이다:
- **할루시네이션 감지:** 출력 모니터링 → 이상 출력 감지 시 Human-in-the-Loop 피드백 수집
- **SLA 관리:** 기업 납품 시 서비스 가용성 약속(SLO/SLA) 위반 시 배상 비용 발생
- **버전 관리:** 모델 업데이트 이력 추적, 이전 버전 롤백 가능한 배포 체계 구축
- **데이터 정합성:** 학습 데이터 품질이 결과 품질을 결정 — 데이터 수집·정제가 핵심

## 6. 장단점 (Pros & Cons)
### 장점 (Pros)
- 도메인 특화 성능 향상 (범용 LLM 대비 특정 태스크에서 우위)
- 데이터 보안 확보 (외부 전송 없음)
- 반복 처리 비용 절감
### 단점 (Cons)
- 초기 학습 데이터 구성 및 GPU 비용 발생
- 범용 능력 저하 (특정 태스크 외 성능 감소)
- 운영 인프라(서빙·모니터링) 직접 관리 필요
