# Prompt Caching (프롬프트 캐싱)

## 1. 개요 (Overview)
- **기술명:** Prompt Caching (프롬프트 캐싱)
- **정의:** LLM API 호출 시 반복적으로 사용하는 긴 프롬프트(시스템 메시지, 문서, 코드베이스 등)를 서버 측에 캐싱하여 재입력 비용과 지연을 줄이는 최적화 기법
- **배경:** [[RAG]]·[[에이전틱 AI]] 워크플로우에서 동일한 문서나 시스템 컨텍스트를 매 호출마다 재전송하는 비용이 문제가 됐다. Anthropic Claude의 캐시 포인트, OpenAI의 Prompt Caching이 대표적 구현이다.

## 2. 주요 특징 (Key Features)
- **비용 절감:** 캐시 히트 시 입력 토큰 비용을 최대 90% 절감 (Anthropic 기준)
- **지연 감소:** 캐시된 토큰은 재처리 없이 즉시 활용 → TTFT(첫 토큰 지연) 감소
- **캐시 유효 시간:** TTL 기반 (Anthropic: 5분, 연장 가능)
- **캐시 포인트 명시:** 개발자가 프롬프트 내 캐시 분기점을 명시적으로 지정

## 3. 작동 원리 (How it Works)
1. 프롬프트에서 자주 반복되는 부분(시스템 프롬프트, 문서 등)에 캐시 포인트 지정
2. 최초 호출 시 해당 구간을 전처리하여 KV 캐시 상태로 서버에 저장
3. 이후 호출 시 동일 캐시 포인트 이전 구간은 재계산 없이 캐시에서 불러옴
4. 캐시 포인트 이후의 새 콘텐츠(사용자 질문 등)만 추가 처리

## 4. 주요 활용 사례 (Use Cases)
- 사례 1: 코딩 어시스턴트 — 전체 코드베이스를 캐싱 후 다양한 질문에 재사용
- 사례 2: RAG 에이전트 — 검색된 문서를 첫 호출에 캐싱, 후속 질문에 재활용
- 사례 3: 롱 시스템 프롬프트 — 복잡한 페르소나·지침을 캐싱하여 매 대화 비용 절감

## 5. 장단점 (Pros & Cons)
### 장점 (Pros)
- 동일 컨텍스트 반복 사용 시 비용·속도 대폭 개선
- 코드 변경 없이 API 레벨에서 적용 가능
### 단점 (Cons)
- TTL 내에만 캐시 유효 — 장기 워크플로우에서 재캐싱 필요
- 캐시 미스 첫 호출은 일반 호출보다 비용이 더 높을 수 있음
