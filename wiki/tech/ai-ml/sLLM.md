# sLLM (Small Large Language Model)

## 1. 개요 (Overview)
- **기술명:** sLLM / SLM (Small (Large) Language Model)
- **정의:** 특정 도메인이나 작업에 특화하여 경량화된 언어 모델로, 파인튜닝을 통해 범용 대형 LLM 없이도 특정 업무에서 충분한 성능을 발휘하도록 설계된 모델
- **배경:** GPT-4 등 대형 클라우드 LLM은 보안·비용·속도 측면에서 기업 실무에 장벽이 있다. 사내 데이터로 특화 파인튜닝한 소형 모델이 특정 작업에서 대형 모델에 준하는 성능을 내면서 실무 도입이 확산됐다.

## 2. 주요 특징 (Key Features)
- **규모:** SLM 0.8B 이하(초경량) ~ sLLM 4B~8B(소형) 범주
- **양자화 적용:** INT4/INT8 양자화로 400MB~4GB 수준까지 압축 → 일반 노트북/소형 서버 실행 가능
- **목적 특화:** 범용 성능보다 특정 태스크(JSON 추출, 분류, 요약 등)에 집중
- **로컬 실행:** Ollama 등을 통해 외부 API 없이 온프레미스 운영
- **비용 구조:** 초기 GPU 구매비용 후 추가 호출 비용 없음 → 대량 처리 시 클라우드 API 대비 경제적
- LoRA 파인튜닝: 16GB 맥북에서도 8B 모델까지 파인튜닝 가능

## 3. 작동 원리 (How it Works)
sLLM 구축 4단계:
1. **베이스 모델 선택:** Qwen, Llama, Mistral 등 오픈소스 경량 모델 선택 (Instruct 버전 권장)
2. **출력 계약 설계:** 모델이 출력해야 할 JSON 스키마, 허용값, 규칙을 프롬프트로 명세
3. **학습 데이터 구성:** (입력 텍스트, 계약된 JSON 출력) 쌍으로 구성된 파인튜닝 데이터셋 작성
4. **LoRA 파인튜닝:** 베이스 모델에 어댑터 추가·학습 (MLX/Unsloth 활용)

## 4. 주요 활용 사례 (Use Cases)
- 사례 1: 비정형 데이터 구조화 파이프라인 — 고객 리뷰, 계약서, 주소 등을 JSON으로 자동 추출
- 사례 2: 보안 민감 환경 — 금융·의료 등 외부 전송 불가 데이터를 온프레미스 sLLM으로 처리
- 사례 3: 반복 업무 자동화 — 하루 수만 건의 문서를 배치 처리하여 인건비 절감
- 사례 4: 채봇 대안 — "무엇이든 물어보세요"형 채봇 대신 특정 업무 자동화 도구로 실무 ROI 창출

## 5. 장단점 (Pros & Cons)
### 장점 (Pros)
- 클라우드 API 비용 독립 (대량 처리 시 장기 총비용 우위)
- 데이터 보안: 외부 서버 전송 불필요
- 빠른 응답: 로컬 실행으로 네트워크 지연 제거
- 목적 특화로 해당 태스크에서 대형 모델에 근접하는 성능
### 단점 (Cons)
- 초기 GPU 투자 비용 발생
- 범용 능력 부족 — 학습한 태스크 외에는 성능 저하
- 모델 업데이트·버전 관리를 직접 수행해야 함
- 인프라(서빙, 모니터링, 장애 대응) 운영 부담

> [!note] 2026-05-12 소스 추가: sLLM 실무 운영 관점
> sLLM은 단순 코드가 아닌 **운영 시스템**이다. 기업 납품 시 SLO/SLA 위반으로 배상 위험이 있으므로 ①할루시네이션 감지 모니터링 ②Human-in-the-Loop 피드백 루프 ③장애 대응·롤백 체계가 필수. 토스 사례처럼 챗봇보다 이커머스 반복 업무 자동화에 먼저 도입하는 것이 ROI 측면에서 효과적. AWS Bedrock·Ollama 등 서빙 인프라 선택도 중요. (출처: 카카오 계열사 AI 개발자 강의)
