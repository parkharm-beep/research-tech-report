# 청킹 (Chunking)

## 1. 개요 (Overview)
- **기술명:** 청킹 (Chunking)
- **정의:** RAG 파이프라인에서 장문 문서를 [[임베딩]] 및 검색에 적합한 크기의 단위(청크)로 분할하는 전처리 전략
- **배경:** LLM의 컨텍스트 윈도우 한계와 임베딩 품질 저하 문제를 해결하기 위해, 문서를 적절한 크기로 나누어 인덱싱하는 방법이 RAG의 필수 단계가 됐다.

## 2. 주요 특징 (Key Features)
- **청크 크기:** 너무 크면 관련 없는 정보 포함, 너무 작으면 맥락 손실 — 보통 256~1024 토큰
- **청크 오버랩:** 인접 청크 간 일부 토큰을 중복 포함하여 경계 단절 방지
- **메타데이터 보존:** 원본 문서명, 페이지 번호, 섹션 제목 등을 청크와 함께 저장

## 3. 주요 청킹 전략 (Strategies)
| 전략 | 특징 |
|---|---|
| **고정 크기 분할** | 일정 토큰 수로 기계적 분할, 빠르고 균일함 |
| **문장·단락 기반** | 언어 구조를 존중하며 분할 |
| **의미 기반 (Semantic)** | 유사 의미의 문장을 묶어 분할 |
| **[[Late 청킹]]** | 먼저 전체 임베딩 후 분할 — 전체 컨텍스트 보존 |
| **[[Contextual Retrieval]]** | 청크마다 원본 문서 맥락 요약을 접두사로 추가 |

## 4. 주요 활용 사례 (Use Cases)
- 사례 1: RAG 파이프라인에서 PDF·웹페이지·보고서 전처리
- 사례 2: 대용량 코드베이스를 함수·클래스 단위로 청킹 후 코드 검색

## 5. 장단점 (Pros & Cons)
### 장점 (Pros)
- 문서 규모에 관계없이 검색 가능한 단위로 분할
- 관련성 높은 청크만 LLM에 전달 → 비용 절감·정확도 향상
### 단점 (Cons)
- 청크 경계에서 문맥 단절 위험
- 최적 청크 크기는 도메인·문서 유형마다 달라 튜닝 필요
