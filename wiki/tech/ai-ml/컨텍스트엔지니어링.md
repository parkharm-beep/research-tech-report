# 컨텍스트 엔지니어링 (Context Engineering)

## 1. 개요 (Overview)
- **기술명:** 컨텍스트 엔지니어링 (Context Engineering)
- **정의:** LLM의 컨텍스트 윈도우에 무엇을 담을지를 설계하는 기술 — 시스템 프롬프트, 검색 결과, 도구 출력, 대화 이력 등을 전략적으로 구성하여 LLM 성능을 극대화하는 접근법
- **배경:** [[프롬프트엔지니어링]]이 단일 프롬프트 문구 최적화에 집중했다면, 컨텍스트 엔지니어링은 에이전트·RAG 시대에 복잡한 컨텍스트 전체를 어떻게 관리할지를 다룬다.

## 2. 주요 특징 (Key Features)
- **컨텍스트 윈도우 최적 활용:** 중요한 정보를 앞·뒤에 배치 (lost-in-the-middle 문제 대응)
- **동적 컨텍스트 구성:** 매 호출마다 필요한 정보만 선별하여 컨텍스트 구성
- **[[Prompt Caching]] 연동:** 재사용 가능한 컨텍스트 구간을 캐시 포인트로 지정
- **압축과 요약:** 긴 대화 이력을 요약하여 컨텍스트 한계 내에서 유지

## 3. 핵심 원칙 (Key Principles)
- **관련성 우선:** 현재 질문에 관련 없는 정보는 컨텍스트에서 제거
- **구조화:** 섹션·헤더·XML 태그로 LLM이 정보를 구조적으로 이해하도록 유도
- **그라운딩:** 사실 기반 정보([[RAG]] 결과)를 프롬프트에 포함하여 할루시네이션 감소
- **지시 명확성:** 태스크·출력 형식·제약 조건을 명확하게 기술

## 4. 주요 활용 사례 (Use Cases)
- 사례 1: RAG 에이전트 — 검색 결과, 이전 단계 출력, 도구 응답을 효율적으로 컨텍스트에 배치
- 사례 2: 멀티턴 챗봇 — 대화 이력을 압축·요약하며 관련 부분만 유지
- 사례 3: 코드 에이전트 — 코드베이스 중 관련 파일만 선별하여 컨텍스트에 포함

## 5. 장단점 (Pros & Cons)
### 장점 (Pros)
- 동일 모델에서 컨텍스트 구성만으로 성능 대폭 향상
- 비용 절감 — 불필요한 토큰 제거
### 단점 (Cons)
- 최적 구성은 도메인·태스크마다 달라 반복 실험 필요
- 동적 컨텍스트 관리는 복잡한 파이프라인 코드 요구
