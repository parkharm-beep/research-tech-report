# 출력 계약 (Output Contract)

## 1. 개요 (Overview)
- **기술명:** 출력 계약 (Output Contract)
- **정의:** LLM이 반드시 따라야 할 출력 형식·스키마·허용 값을 프롬프트에 계약서처럼 명세하여, 파싱 가능하고 검증 가능한 구조화 출력을 강제하는 프롬프트 설계 기법
- **배경:** 자연어 형태의 LLM 응답은 DB 저장·자동화 파이프라인 연결이 불가능하다. 비정형 데이터 구조화, 계약서 분석 등 실무 자동화에서는 JSON 등 정형 출력이 필수이며, 프롬프트로 출력 형식을 강제하는 패턴이 실무 표준으로 자리잡았다.

## 2. 주요 특징 (Key Features)
- 시스템 프롬프트에 JSON 키 이름, 타입, 허용값, 필수 여부를 명시
- "반드시 단 하나의 JSON 오브젝트만 출력"과 같은 강제 규칙 포함
- Pydantic 등 타입 검증 라이브러리로 출력 유효성을 코드 레벨에서 재검증
- 검증 실패 시 Repair Prompt로 재시도(retry) 가능
- 천 줄의 정규식 대신 열 줄의 잘 설계된 프롬프트로 대체

## 3. 작동 원리 (How it Works)
출력 계약의 3요소:
1. **키 이름:** 일관된 snake_case, 의미 명확한 필드명 (예: `sentiment`, `issue_type`)
2. **타입 및 허용값:** `str`, `int`, `Literal["positive","negative","neutral"]` 등 명시
3. **필수 여부:** Required / Optional 분리

프롬프트 구조 예시:
```
System: 당신은 고객 리뷰 분석 엔진입니다.
반드시 아래 JSON 스키마만 출력하시오:
{"sentiment": "positive|negative|neutral", "issue": "str", "delay_days": int|null}
절대 다른 텍스트를 포함하지 마시오.

User: 다음 리뷰를 분석하세요: {review_text}
```

## 4. 주요 활용 사례 (Use Cases)
- 사례 1: 고객 리뷰 자동 분류 — 감정, 이슈 유형, 배송 지연일 등을 JSON으로 추출하여 대시보드화
- 사례 2: 계약서 위험 조항 검토 — 위험 조항 유형·심각도를 구조화된 테이블로 출력
- 사례 3: 주소 정제 — 비정형 주소 텍스트를 표준 행정 구역 체계로 정규화
- 사례 4: 악성 로그 분류 — 로그 텍스트에서 위협 유형·권장 조치를 구조화 추출

## 5. 장단점 (Pros & Cons)
### 장점 (Pros)
- 자동화 파이프라인(DB 저장, 대시보드 연동)에 직접 연결 가능
- 정규식 기반 파싱보다 표현 다양성(동의어, 어순 변형)에 강건
- Pydantic 검증 + Repair Prompt 조합으로 출력 신뢰도 향상
- 추출 정확도를 수치(예: 95% 이상)로 목표 설정 → QA 기준 수립 가능
### 단점 (Cons)
- 출력 계약 설계 자체가 도메인 이해를 전제하므로 초기 설계 비용 존재
- 모델이 계약을 어기는 경우(혼합 출력, 형식 불일치) 재시도 비용 발생
- Temperature가 높으면 형식 안정성이 떨어지므로 추출 작업은 낮은 Temperature 필수
